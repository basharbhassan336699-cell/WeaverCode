5. **لوحة الويب 📊**: قسم جديد يعرض حالة وضع التخطيط (تبديل)، الخطة المعلّقة

## v4.9.0 — اكتشاف النماذج في الطرفية + مزامنة ثنائية الطرفية ↔ الويب

الطرفية كانت «تبقى معلّقة على نماذج المنصة السابقة» ولا تتزامن مع الويب لأنها
تقرأ `os.environ` المحمّل عند الإقلاع فقط، و`/model` لم يكن يكتب `.env`. الإصلاح
(228 اختباراً ناجحة) — المصدر الوحيد للحقيقة = `config/.env`:

  قبل كل طلب في الحلقة التفاعلية وقبل اكتشاف النماذج — فتغييرات الويب تظهر فوراً
  في الطرفية (لا تبقى على المنصة السابقة).
- **`save_env()`**: أوامر الطرفية تكتب `.env` فيلتقطها الويب تلقائياً.
- **`_sync_provider_from_env()`**: يُحدِّث `provider.config` الحيّ (مفتاح/رابط/نموذج)
  دون إعادة تشغيل.
- **أوامر جديدة في الطرفية**: `/key <k>` (تغيير المفتاح)، `/provider <name> [key]`
  (تبديل المنصة + اكتشاف نماذجها فوراً)، `/models` (اكتشاف نماذج المنصة الحالية).
  `/model` صار يحفظ في `.env` (يُزامَن مع الويب).

## v4.7.0 — العمل الفعلي على مستودعات GitHub (استنساخ حقيقي) + رد أوضح

**السبب الجذري لـ «فتحت مستودعاً ولم يفعل شيئاً»:** كان اختيار المستودع تجميلياً —
يضيف سطر سياق فقط، والمستودع **لا يُستنسَخ**، ومجلد عمل الوكيل = المخرجات لا
المستودع. فالنموذج يخترع مساراً وهمياً (`…/Weaver-Write/weaver.py`)، يفشل، ولا
ينتج شيئاً. الإصلاح (219 اختباراً ناجحة):

1. **استنساخ حقيقي**: اختيار مستودع يستدعي `/api/github/select-repo` الذي يستنسخه
   محلياً (بمصادقة التوكن، بلا تخزينه في `.git/config`) ويسجّله مساحة عمل نشِطة
   في `config/workspace.json`. الرقاقة تعرض عدد الملفات المستنسَخة.
2. **مجلد عمل الوكيل = المستودع**: الـ daemon يقرأ `workspace.json` قبل كل مهمة
   (`_active_work_dir`) فيعمل الوكيل على ملفات المستودع الحقيقية بمسارات نسبية.
3. **رفع للمستودع الصحيح**: `_github_push` يعمل على مساحة العمل النشِطة (حقن التوكن
   وقت الرفع فقط). «محادثة بدون مستودع» تلغي مساحة العمل (`/api/workspace/clear`).
4. **رد أوضح عند غياب النص**: إن نفّذ الوكيل أدوات دون كتابة نص ختامي، يُعرض ملخّص
   ما فُعل بدل JSON خام؛ وإن لم ينفّذ شيئاً، إرشاد لسبب محتمل (سياق كبير/طلب غامض).

- **3xx لا يُعاد على مستوى curl** (كان يُضاعف الطلبات) — يتكفّل به سلّم
  complete() فقط.
- **ذاكرة تكيّف المزوّد** (`~/.weaver/provider_cache.json`): يُحفظ ما تعلّمناه
  (وضع أدنى/صيغة/إسقاط أدوات/max_tokens) لكل مزوّد، فلا يُعاد الاكتشاف بعد
  إعادة التشغيل. النتيجة: اكتشاف لمرة واحدة (بضع طلبات)، ثم **طلب واحد لكل
  رسالة دائماً** — أُثبِت بمحاكاة إعادة التشغيل.

ملاحظة صريحة: إن ظلّت البوابة ترفض حتى الطلب الأدنى، فهي مقيّدة بما يتجاوز
قدرة أي كود؛ مزوّد يدعم الطلبات القياسية (Groq/الرسمي) يعمل بكامل القدرات.

## v3.0.1 — مطابقة طلب aerolink الناجح + إصلاح معالجة 305

كشف اختبار المستخدم اليدوي (curl مباشر) أن aerolink **يعمل ويردّ بنص حقيقي**،
بينما كان WeaverCode يحصل على `HTTP 305 «Service Unavailable»`. الفرقان:

- **الترويسات**: كان WeaverCode يرسل `x-api-key` **و** `Authorization` معاً،

  يدوياً)، وAnthropic الرسمي وحده يحصل على `x-api-key`.
- **خطأ معالجة 305**: كانت الحالة 305 (وكل 3xx) تُعامَل كنجاح `< 400` فتفشل
  عند تحليل JSON وتظهر «استجابة ليست JSON صالحاً». الآن 3xx تُرفع كخطأ
  **عابر واضح** فيعيد WeaverCode المحاولة تلقائياً (غالباً ينجح كما نجح يدوياً).

  - **مؤشر تفكير متحرك** 🕸️/🕷️ عبر صنف `Spinner` غير متزامن يعمل أثناء
    انتظار رد المزود، ويُفرغ سطره تلقائياً عند طباعة استدعاءات الأدوات.
  - عرض موحّد للأدوات والردود والأخطاء والإحصاءات، وسطر إدخال أنيق `❯`.
  - قراءة الإعدادات من متغيرات البيئة، واحترام `WEAVER_DB_PATH`، وتفادي مسح
    الشاشة عند التمرير عبر أنبوب (غير تفاعلي)، مع الحفاظ على التوافق الرجعي
    لأسماء الدوال القديمة.
- **`weaver.py`** — ربط الواجهة الجديدة: شاشة الاستقبال عند بدء أي وضع،
  رأس الشاشة المنقسمة، ومؤشر التفكير المتحرك في وضعَي المهمة الواحدة والتفاعلي.

## v1.3.0 — دعم مزدوج للمزودين ونقل عبر curl

### أُصلح / تغيّر
- **`core/engine/provider.py`** — إعادة كتابة كاملة لمحرك الاتصال:
  - دعم صيغة **Anthropic** (`POST /v1/messages`) تلقائياً عند اكتشاف
    `aerolink` أو `anthropic` في `WEAVER_BASE_URL`.
  - دعم صيغة **OpenAI** (`POST /chat/completions`) لبقية المزودين
    (OpenRouter، Groq، DeepSeek، OpenAI، Ollama...).
  - استخدام **curl** داخلياً بدل httpx لتجاوز مشاكل إعادة التوجيه (redirect)
    والحالة 305، مع خيارات `follow_redirects` و`--location-trusted`.
  - دعم **بروكسي اختياري** عبر `WEAVER_PROXY` أو `HTTPS_PROXY`/`HTTP_PROXY`.
  - **توحيد الاستجابة**: يُحوَّل رد Anthropic إلى شكل OpenAI (`choices[0].message`)
    فيبقى `query_engine` موحّداً دون تغيير، مع تحويل مخطط الأدوات والرسائل
    (system/tool_use/tool_result) بين الصيغتين.
  - **معالجة كاملة للأخطاء برسائل عربية واضحة** لكل حالات HTTP الشائعة
    (401, 403, 404, 305, 307, 308, 429, 500, 503) وأخطاء الشبكة وJSON.
  - إجبار الصيغة يدوياً عبر `WEAVER_API_FORMAT=anthropic|openai`.
- **`prompts/system.py`** — تشديد قاعدة الهوية: يعرّف الوكيل عن نفسه كـ
  WeaverCode فقط، ولا يذكر أي نموذج أو شركة أو منصة (لا إثباتاً ولا نفياً)،
  والعربية هي اللغة الافتراضية.
- **`config/.env.example`** — شرح الفرق بين صيغتي OpenAI وAnthropic، وإضافة
  **aerolink** كمثال جاهز لمزوّد متوافق مع Anthropic
  (`WEAVER_BASE_URL=https://capi.aerolink.lat`، `WEAVER_MODEL=claude-fable-5`)،
  وإضافة متغيرات `WEAVER_API_FORMAT` و`WEAVER_PROXY` و`WEAVER_FOLLOW_REDIRECTS`
  و`WEAVER_TIMEOUT`.
- **`weaver.py`** — تحميل `config/.env` تلقائياً عند بدء التشغيل، وتحسين قارئ
  `.env` (دعم `export` وإزالة علامات الاقتباس). البانر يعرض اسم WeaverCode فقط.

### ملاحظة
- الآن `python3 weaver.py "قل مرحباً"` يعمل مع أي مزوّد
  (aerolink / groq / openrouter / anthropic / openai) بمجرد ضبط المفتاح في `.env`.
